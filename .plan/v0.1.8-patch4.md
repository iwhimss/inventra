## Bu Sefer Farklı: Tahmin Değil, Kaynak Kodda Doğrulanmış Kanıt

Önceki üç turda (`v0.1.8-patch`, `patch2`, `patch3`) `audioplayers` paketinin **davranışını dokümantasyondan ve genel bilgiden çıkarım yaparak** tahmin ediyordum — her tur bir önceki turun eksik bıraktığı katmanı buluyor ama yeni bir tahmine dayanıyordu. Kullanıcının bu son bulgusu ("her zaman aynı: ilk ürün çalıyor, sonrakiler hiç çalmıyor, restart bile örüntüyü değiştirmiyor") o kadar **deterministik ve tekrarlanabilir** ki, bu sefer paketin **Android native (Kotlin) kaynak kodunu doğrudan okuyarak** kesin kanıt aradım — ve buldum.

**Kaynak dosya:** `audioplayers_android` paketi → `SoundPoolPlayer.kt` (bu, `PlayerMode.lowLatency` seçildiğinde devreye giren native oynatıcı sınıfı — v0.1.8-patch2'de geçtiğimiz mod):

```kotlin
override fun start() {
    val streamId = streamId
    val soundId = soundId

    if (streamId != null) {
        soundPool.resume(streamId)              // ← SORUN BURADA
    } else if (soundId != null) {
        this.streamId = soundPool.play(soundId, ...)   // gerçek yeni çalma
    }
}

override fun stop() {
    streamId?.let {
        soundPool.stop(it)
        streamId = null                          // ← streamId yalnızca burada sıfırlanıyor
    }
}
```

**Kesin kök neden:** `AudioPlayer.resume()` çağrısı (Dart tarafında bizim her tetiklemede çağırdığımız metod), native tarafta yukarıdaki `start()` fonksiyonuna denk geliyor. Mantık şu: **`streamId` daha önce hiç ayarlanmadıysa** (`null`), `soundPool.play(...)` çağrılır — bu gerçek, yeni bir ses çalma işlemidir ve `streamId`'yi doldurur. **Ama `streamId` bir kez dolduktan sonra**, sonraki her `resume()` çağrısı artık `soundPool.resume(streamId)` dalına düşer — bu, Android SDK'sının **duraklatılmış (paused) bir akışı devam ettirme** metodudur, **bitmiş/tamamlanmış bir akışı yeniden başlatmaz**. `streamId`, yalnızca `stop()` çağrıldığında `null`'a sıfırlanıyor (yukarıdaki `stop()` fonksiyonu).

**Bizim v0.1.8-patch3'teki hatamız:** O turda, "SoundPool modunda `stop()` gerekli değil, hatta üst üste binmeyi engelleyebilir" diye düşünüp `_play()`'den `await player.stop();` satırını **kaldırmıştım**. Kaynak koddan görüldüğü üzere bu **tam tersine kritikmiş**: `stop()` çağrılmadığı sürece `streamId` hiç sıfırlanmıyor, bu yüzden **ilk çalmadan sonraki her `resume()` çağrısı sessizce hiçbir şey yapmayan `soundPool.resume(streamId)` dalına düşüyor** — tam olarak kullanıcının gözlemlediği "ilk ürün çalıyor, sonrakiler asla çalmıyor, restart bile değiştirmiyor (çünkü restart sonrası yine ilk çalma `play()` dalına düşüyor, ikincisi yine `resume()`'un ölü dalına düşüyor)" davranışı.

## Çözüm

Basit ve kesin: `_play()`'e `player.stop()` çağrısını **`resume()`'dan hemen önce** geri ekliyoruz. Bu, native `streamId`'yi her tetiklemeden önce `null`'a sıfırlayıp bir sonraki `resume()` çağrısının **doğru dala** (`soundId != null` → `soundPool.play(...)` → gerçek yeni çalma) düşmesini garanti ediyor:

```dart
static Future<void> _play(SoundCategory category, bool enabled, double volume) async {
  if (!enabled) return;
  final now = DateTime.now();
  final last = _lastPlayTimes[category];
  if (last != null && now.difference(last) < const Duration(milliseconds: 100)) return;
  _lastPlayTimes[category] = now;

  try {
    final player = await _getPlayer(category);
    await player.stop();               // streamId'yi sıfırlar — resume() sonraki tetiklemede
                                        // soundPool.play() dalına düşsün, ölü resume() dalına değil
    await player.setVolume(masterVolume * volume);
    await player.resume();
  } catch (_) {
    final broken = _players.remove(category);
    _loaded.remove(category);
    await broken?.dispose();
  }
}
```

**Bu, v0.1.8-patch2'deki `stop()`'u geri koymakla aynı satır ama artık NEDENİ kesin biliniyor** (o zamanki "üst üste binen çağrıyla yarışıyor olabilir" varsayımı yanlıştı — gerçek neden, SoundPool'un `streamId` durum makinesiydi). `setSourceAsset()` (kaynağı bir kez yükleme, v0.1.8-patch3'te eklendi) **doğruydu ve korunuyor** — `stop()`, kaynağı YENİDEN YÜKLEMİYOR, yalnızca `streamId`'yi sıfırlayıp bir sonraki `play()` çağrısının önceden yüklenmiş `soundId`'yi kullanarak hızlıca tetiklenmesini sağlıyor. Yani kaynağın tekrar tekrar yüklenmesi riski (v0.1.8-patch'teki asıl sorun) geri gelmiyor; bu sefer hem "bir kez yükle" hem "her seferinde doğru şekilde tetikle" bir arada.

**Güven seviyesi:** Bu tur, önceki üçünden farklı olarak paketin **gerçek native kaynak kodunda satır satır doğrulanmış** bir bulguya dayanıyor — tahmin değil. Yine de gerçek cihazda son testi kullanıcı yapacak.
