## Kök Neden Analizi

### [B1] + [B2] — Aynı Kök Neden: Yanlış Ses Motoru Modu

**Neden önceki iki düzeltme (kategori bazlı debounce, "ısıtma") yetersiz kaldı:** İkisi de doğru gözlemlere dayanıyordu ama **yanlış katmanı** düzeltiyordu. `sound_service.dart`, her ses için varsayılan `AudioPlayer` modunu (`PlayerMode.mediaPlayer`) kullanıyor — bu mod Android'de **ExoPlayer/MediaPlayer** alt yapısını kullanır. Bu altyapı, **uzun/karmaşık ses dosyaları** (müzik, podcast) için tasarlanmıştır ve her `play()` çağrısında görece ağır bir hazırlık süreci (native nesne oluşturma, buffer ayırma, bazen arka plan thread senkronizasyonu) içerir. Kısa UI bildirim sesleri (birkaç yüz milisaniyelik `.wav` dosyaları) için bu altyapı **yanlış araç**tır:

- **"Isıtma" neden kalıcı çözmedi:** Açılışta bir kez ısıtma yapılsa bile, `AudioPlayer` bir süre kullanılmayınca (ör. kullanıcı Ürünler sekmesinde ürün ararken birkaç saniye geçiyor), Android bazı cihazlarda/sürümlerde **arka planda boşta duran MediaPlayer oturumlarını** kaynak tasarrufu için geri talep edebiliyor. Sekmeye her yeniden girişte gözlenen "ilk ürün sessiz, ikincisi çift" paterni, tam olarak bu tür bir **yeniden ısınma** döngüsünün tekrarlanmasıyla açıklanıyor — ısıtma yalnızca *uygulama açılışında* yapıldığı için sekme geçişleri sonrası oluşan yeni "soğuma" durumlarını kapsamıyor.
- **Barkod ile eklemede çalmaması:** `_showPriceSelectionDialog()` (sesin merkezi tetikleyicisi) barkod tarama akışında da çağrılıyor — kod satırı doğru. Ancak barkod tarama sayfası kapanırken (`Navigator.pop()`) çalışan sayfa geçiş animasyonu ile **aynı anda** MediaPlayer tabanlı `play()` çağrısı tetikleniyor; bu ağır native hazırlık süreci, yoğun bir animasyon karesiyle çakışınca bazı cihazlarda sessizce atlanabiliyor/gecikip fark edilmeyebiliyor.

**Doğru çözüm — `PlayerMode.lowLatency` (Android'de SoundPool):** `audioplayers` paketi, tam olarak bu senaryo (oyun/UI ses efektleri — kısa, sık tetiklenen, bazen üst üste binen sesler) için ayrı bir mod sunuyor: `PlayerMode.lowLatency`. Bu mod Android'de **SoundPool** API'sini kullanır — ses dosyası belleğe **önceden decode edilip yüklenir**, tetikleme anında **native tarafta neredeyse gecikmesiz** çalar, ve **üst üste binen çağrıları doğal olarak destekler** (bir önceki sesi durdurmaya gerek kalmadan). Bu, hem "soğuma" sorununu (SoundPool oturumu MediaPlayer gibi ağır değil, tekrar tekrar geri talep edilme riski çok daha düşük) hem de "çift çalma/kaçırma" sorununu (artık `stop()` ile yarışan bir `play()` çağrısı yok) kökten çözer.

**Uygulama planı:**
1. `_getPlayer()` içinde her `AudioPlayer` oluşturulduğunda `player.setPlayerMode(PlayerMode.lowLatency)` çağrılır.
2. `_play()` içindeki `await player.stop();` satırı kaldırılır — SoundPool modunda gerekli değil, hatta üst üste binen doğal çalmayı engelleyebilir.
3. Mevcut "ısıtma" adımı (`_warmUpPlayers`) **korunur** — SoundPool'a sesin önceden yüklenmesini (decode) sağlamak için hâlâ faydalı, sadece artık "soğuma" riski büyük ölçüde ortadan kalkacağı için tek seferlik ısıtma yeterli olacak.
4. Kategori bazlı debounce (v0.1.8) korunur — SoundPool ile de çok hızlı ardışık tetiklemelerde (ör. yanlışlıkla çift dokunma) gereksiz üst üste binmeyi önlemeye devam eder.

**Not:** `PlayerMode.lowLatency`'nin Android dışı platformlarda (Windows, iOS) etkisi sınırlı olabilir (SoundPool Android'e özgüdür); ancak bu uygulamanın birincil şikayet kaynağı mobil (Android) olduğu için asıl fayda orada görülecek. Diğer platformlarda davranış en kötü ihtimalle mevcutla aynı kalır, kötüleşmez.

### [B3] Barkod İptal — Adaptif Eşik Tasarımı

**Önceki iki turun neden başarısız olduğu (geriye dönük analiz):**
- **v0.1.8 (250ms):** Kameranın gerçek algılama aralığı (ardışık başarılı okumalar arası boşluk) bu değerden **daha uzun** olabiliyor (analiz sıklığı sınırlı, otomatik odaklanma, hafif titreme) — barkod hareketsiz tutulsa bile yanlışlıkla iptal ediliyordu.
- **v0.1.8-patch (600ms):** Bu sefer onay süresine (700ms) **çok yakın** bir değer seçildi — aradaki marj yalnızca 100ms. Barkod, onay penceresinin **geç bir anında** (ör. 400-500ms civarında) çıkarılırsa, watchdog'un bunu fark edip iptal etmesi için kalan süre onay tamamlanmadan yetişmeye yetmiyor — bu yüzden "eskisi gibi hiç iptal olmuyor" hissi oluştu.

**Kök sorun:** Doğru eşik değeri **cihaza göre değişiyor** (kamera donanımı, ışık koşulu, `mobile_scanner`'ın o anki analiz hızı). Sabit bir sayı ile iki farklı gereksinimi (yanlış-pozitifi önlemek İÇİN yeterince gevşek, ama onay süresinden önce tepki verebilmek İÇİN yeterince sıkı) aynı anda karşılamak, cihazdan cihaza değişen bir değişkenle mümkün değil.

**Adaptif çözüm:** Sabit bir eşik yerine, **son birkaç başarılı algılama arasındaki gerçek boşluğun ortalaması** canlı ölçülür ve iptal eşiği bu ortalamanın bir katı (3x) olarak **otomatik** hesaplanır — cihazın kendi ritmine göre kendini ayarlar:

```dart
final List<int> _recentGapsMs = [];  // son 5 algılama arası boşluk (ms)
DateTime? _lastSeenAt;
Timer? _watchdog;

// Güvenlik sınırları: eşik ne çok küçük (yanlış-pozitif) ne de onay
// süresine (700ms) çok yakın (yetişememe riski) olmamalı.
static const _minThreshold = Duration(milliseconds: 250);
static const _maxThreshold = Duration(milliseconds: 500); // 700ms onaydan güvenli marj

void _onDetect(BarcodeCapture capture) {
  if (capture.barcodes.isEmpty) { _cancelCandidate(); return; }
  final code = capture.barcodes.first.rawValue;
  if (code == null || code.isEmpty) { _cancelCandidate(); return; }

  final now = DateTime.now();
  if (_lastSeenAt != null) {
    _recentGapsMs.add(now.difference(_lastSeenAt!).inMilliseconds);
    if (_recentGapsMs.length > 5) _recentGapsMs.removeAt(0);
  }
  _lastSeenAt = now;
  _watchdog ??= Timer.periodic(const Duration(milliseconds: 100), (_) => _checkWatchdog());

  if (_candidateCode != code) {
    setState(() => _candidateCode = code);
    _confirmController..stop()..reset()..forward();
    _recentGapsMs.clear(); // yeni aday için geçmiş ritim verisini sıfırla
  }
}

void _checkWatchdog() {
  if (_candidateCode == null || _lastSeenAt == null) return;
  final avgGapMs = _recentGapsMs.isEmpty
      ? 150 // henüz veri yoksa makul bir varsayılan
      : _recentGapsMs.reduce((a, b) => a + b) ~/ _recentGapsMs.length;
  final threshold = Duration(
    milliseconds: (avgGapMs * 3).clamp(_minThreshold.inMilliseconds, _maxThreshold.inMilliseconds),
  );
  if (DateTime.now().difference(_lastSeenAt!) > threshold) _cancelCandidate();
}
```

**Neden bu çalışır:**
- Hızlı algılayan bir cihazda (ör. ortalama 80ms'de bir okuma) eşik `80×3=240ms` → alt sınıra (`250ms`) yakın, **hızlı iptal** — barkod çıkarılır çıkarılmaz neredeyse anında fark edilir.
- Yavaş/titrek algılayan bir cihazda (ör. ortalama 200ms) eşik `200×3=600ms` → üst sınıra (`500ms`) çarpar ve orada sabitlenir — **yanlış-pozitif olmaz**, ama yine de 700ms onaydan önce (200ms marjla) tepki verme garantisi korunur.
- `_maxThreshold` (500ms), onay süresinden (700ms) **her zaman** en az 200ms kısa tutularak, watchdog'un onay tamamlanmadan önce **her koşulda** devreye girme şansı garanti edilir — v0.1.8-patch'teki yarış durumu (600ms vs 700ms, yalnızca 100ms marj) bir daha yaşanmaz.
- Aday barkod değiştiğinde (`_recentGapsMs.clear()`) geçmiş ritim verisi sıfırlanır — bir önceki barkodun ritmi, yeni bir barkodun değerlendirmesini yanıltmaz.
