## Önce Dürüstlük: Buraya Nasıl Geldik

Bu, ses ve barkod tarama konusunda **üçüncü düzeltme turu**. Önceki iki turda (v0.1.8-patch, v0.1.8-patch2) yaptığım tanılar kısmen doğruydu ama her seferinde **bir sonraki katmandaki** asıl sorunu gözden kaçırdım:
- 1. tur: "soğuk başlangıç" → ısıtma eklendi → yetmedi.
- 2. tur: "yanlış ses motoru modu" → `PlayerMode.lowLatency`'ye geçildi → **daha kötü** bir duruma yol açtı (ilk kullanımdan sonra kalıcı sessizlik, sadece tam uygulama kapat-aç düzeltiyor).

Bu son bulgu çok önemli bir ipucu: "tam kapat-aç düzeltiyor ama sayfa değiştirmek düzeltmiyor" — bu, **native (platform tarafı) bir kaynağın kalıcı olarak bozulduğu ve yalnızca işlem tamamen öldüğünde sıfırlandığı** anlamına geliyor. Bu sefer kök nedeni, paketin **kendi kaynak kodundaki açık uyarısını** okuyarak buldum (aşağıda).

Barkod tarama tarafında ise kullanıcı zaten doğru teşhisi koydu: zamanlama tabanlı (sabit süre, sonra adaptif süre) bir "iptal" mekanizması kurmaya çalışmak üç turdur güvenilir çalışmıyor — bu yüzden **mekanizmanın kendisini kaldırıp** çok daha basit ve öngörülebilir bir tasarıma geçiliyor.

## Kök Neden Analizi

### [B1] Sepete Ekleme Sesi — Gerçek Kök Neden Bulundu

**`audioplayers` paketinin kendi kaynak kodunda (`audioplayer.dart:193-194`), `play()` metodunun dokümantasyonunda şu not var:**
> "To reduce preparation latency, instead consider calling `setSource` beforehand and then `resume` separately."

Yani paketin kendisi, **her tetiklemede `play(AssetSource(...))` çağırmanın yanlış kullanım deseni olduğunu** açıkça belirtiyor. Bizim kodumuz (üç turdur) tam olarak bunu yapıyordu: her `_play()` çağrısında `player.play(AssetSource(asset))` — bu, kaynağı **her seferinde yeniden yüklüyor** (asset bundle'dan okuma + native tarafta yeniden hazırlama).

**`PlayerMode.lowLatency`'ye geçtiğimizde bu yanlış kullanım deseni çok daha ciddi bir soruna dönüştü:** Bu mod Android'de **SoundPool** kullanır — SoundPool, sesleri sınırlı sayıda "slot"a önceden yükleyip oradan çalacak şekilde tasarlanmıştır. Her `play()` çağrısında kaynağı **yeniden yüklemek**, her seferinde SoundPool'da **yeni bir slot** talep etmek anlamına geliyor; eski slotlar düzgün serbest bırakılmadığı için birkaç çağrı sonra SoundPool'un **native kaynak limiti** doluyor ve o andan itibaren **tüm çalma istekleri sessizce başarısız oluyor** (`catch (_) {}` bunu yutuyor). Bu native kaynak durumu yalnızca **uygulama süreci tamamen sonlandırıldığında** sıfırlanıyor — kullanıcının "sayfayı kapatıp açmak yetmiyor, tam uygulamayı kapatmam lazım" gözleminin birebir açıklaması bu.

**Doğru çözüm — kaynağı bir kez yükle, sonra yalnızca `resume()` ile tetikle:**
```dart
// Kategori başına SABİT asset — tek doğruluk kaynağı
static const _assetPaths = {
  SoundCategory.success: 'sounds/success.wav',
  SoundCategory.error: 'sounds/error.wav',
  SoundCategory.warning: 'sounds/notification.wav',
  SoundCategory.notification: 'sounds/notification.wav',
  SoundCategory.info: 'sounds/success.wav',
  SoundCategory.login: 'sounds/login.wav',
  SoundCategory.cartAdd: 'sounds/cart_add.wav',
};
static final Set<SoundCategory> _loaded = {};

static Future<AudioPlayer> _getPlayer(SoundCategory category) async {
  var player = _players[category];
  if (player == null) {
    player = AudioPlayer();
    player.setReleaseMode(ReleaseMode.stop);
    await player.setPlayerMode(PlayerMode.lowLatency);
    _players[category] = player;
    _loaded.remove(category); // yeni oyuncu — kaynak henüz yüklenmedi
  }
  if (!_loaded.contains(category)) {
    try {
      await player.setSourceAsset(_assetPaths[category]!);
      _loaded.add(category);
    } catch (_) {}
  }
  return player;
}

static Future<void> _play(SoundCategory category, bool enabled, double volume) async {
  if (!enabled) return;
  final now = DateTime.now();
  final last = _lastPlayTimes[category];
  if (last != null && now.difference(last) < const Duration(milliseconds: 100)) return;
  _lastPlayTimes[category] = now;
  try {
    final player = await _getPlayer(category);
    await player.setVolume(masterVolume * volume);
    await player.resume();       // ← play() DEĞİL, kaynak zaten yüklü
  } catch (_) {
    // Kendi kendini onarma: native oyuncu bozulmuş olabilir — at, bir
    // sonraki çağrıda sıfırdan kurulup kaynağı yeniden yüklensin. Artık
    // tam uygulama yeniden başlatmaya asla gerek kalmıyor.
    final broken = _players.remove(category);
    _loaded.remove(category);
    await broken?.dispose();
  }
}
```

Bu tasarımın iki katmanı var:
1. **Kök neden düzeltmesi:** Kaynak yalnızca bir kez (`setSourceAsset`) yüklenir; tetikleme yalnızca `resume()` ile yapılır — SoundPool slotu tükenmesi bir daha oluşmaz.
2. **Kendi kendini onarma güvenlik ağı:** `resume()` her ne sebeple olursa olsun (bilinmeyen bir platform tuhaflığı) başarısız olursa, o kategorinin oyuncusu atılıp bir sonraki çağrıda sıfırdan kurulur — native bir bozulma **bir daha asla tam uygulama yeniden başlatması gerektirmeyecek**, en kötü ihtimalle bir sonraki tıklamada kendi kendine düzelecek.

`playSuccess()`, `playError()` vb. genel metodlar artık `asset` parametresi taşımıyor (tek doğruluk kaynağı `_assetPaths` haritası) — yanlışlıkla farklı bir string yazılıp senkron dışı kalma riski de ortadan kalkıyor.

### [B2] Barkod Tarama — Zamanlama Tabanlı Mekanizma Tamamen Kaldırılıyor

**Kullanıcının teşhisi doğru:** Üç turdur (sabit 250ms, sabit 600ms, adaptif 250-500ms) "ne kadar süre onay bekleyelim / ne zaman iptal edelim" sorusuna zamanlamaya dayalı bir cevap bulmaya çalıştık ve hiçbiri gerçek dünyada güvenilir çalışmadı — çünkü `mobile_scanner`'ın gerçek algılama davranışı (callback sıklığı, sessiz kalma durumları) cihazdan cihaza öngörülemeyecek kadar değişken. Bu mekanizmayı daha fazla ince ayarla kurtarmaya çalışmak yerine **komple kaldırılıp** çok daha basit, öngörülebilir bir modele geçiliyor:

**Yeni tasarım — "başlangıç bekleme + anında tarama":**
1. Kamera sayfası açılır açılmaz **0.5 saniyelik bir "hazır değil" penceresi** başlar — bu sırada `onDetect` callback'leri tamamen yok sayılır. Bu, kullanıcının kamerayı barkoda **doğrultma/konumlandırma anında** yanlışlıkla bir şey taratmasını önler (en yaygın "istenmeyen tarama" senaryosu tam olarak budur).
2. 0.5 saniye dolduktan sonra, çerçeve içinde **ilk başarılı okuma anında** barkod **anında** kabul edilir — onay bekleme, watchdog, "son görülme" takibi gibi hiçbir zamanlama mantığı yok.
3. Yanlış barkod okuma riski, zamanlama yerine **iki basit, güvenilir önlemle** azaltılır:
   - Çerçeve **hafifçe küçültülür** (%65×%28 → %60×%25 — kullanıcı isteğiyle aşırı küçültme yapılmıyor, ölçülü bir daraltma) — kamera görüş alanında yanlışlıkla komşu bir barkodun çerçeveye girme ihtimali azalır.
   - `scanWindow` zaten yalnızca çerçeve içindeki alanı algılıyor (v0.1.5-patch2'den beri) — küçültülmüş çerçeveyle bu kısıtlama daha da isabetli hale gelir.

**Neden bu daha iyi bir çözüm:** Zamanlama tabanlı yaklaşımların tümü "ne kadar bekleyelim" sorusuna cihaz-bağımsız bir cevap bulmaya çalışıyordu ve bu soru pratikte cevaplanamaz olduğu kanıtlandı. "Anında tara ama önce kısa bir yerleştirme süresi tanı + çerçeveyi daraltarak isabeti artır" yaklaşımı, zamanlamaya hiç güvenmediği için **hiçbir cihazda "yanlış ayarlanmış eşik" riski taşımıyor** — ya çerçevede bir barkod var ya yok, üçüncü bir "ne kadar süre emin olalım" belirsizliği kalmıyor.

**Kod tarafında büyük sadeleşme:** `_confirmController`, `_candidateCode`, `_watchdog`, `_recentGapsMs`, `_lastSeenAt`, `_checkWatchdog()`, `_cancelCandidate()`, `_confirmScan()` — hepsi kaldırılıyor. Ekrandaki "Taranıyor: xxx" onay göstergesi de kaldırılıp sabit "Barkodu çerçeveye yerleştirin" metnine dönülüyor.

```dart
class _BarcodeScannerPageState extends State<BarcodeScannerPage> with TickerProviderStateMixin {
  final MobileScannerController _controller = MobileScannerController();
  late final AnimationController _scanLineController;
  late final Animation<double> _scanLineAnim;
  bool _readyToScan = false;
  bool _handled = false;
  Timer? _startupGraceTimer;

  @override
  void initState() {
    super.initState();
    _scanLineController = AnimationController(vsync: this, duration: const Duration(milliseconds: 1800))..repeat(reverse: true);
    _scanLineAnim = CurvedAnimation(parent: _scanLineController, curve: Curves.easeInOut);
    _startupGraceTimer = Timer(const Duration(milliseconds: 500), () {
      if (mounted) setState(() => _readyToScan = true);
    });
  }

  @override
  void dispose() {
    _scanLineController.dispose();
    _startupGraceTimer?.cancel();
    _controller.dispose();
    super.dispose();
  }

  void _onDetect(BarcodeCapture capture) {
    if (!_readyToScan || _handled) return;
    if (capture.barcodes.isEmpty) return;
    final code = capture.barcodes.first.rawValue;
    if (code == null || code.isEmpty) return;
    _handled = true; // Navigator.pop() tamamlanana kadar tekrar tetiklenmeyi önler
    Navigator.of(context).pop();
    widget.onDetected(code);
  }
  ...
}
```

`_handled` bayrağı, sayfa kapanış animasyonu sürerken gelebilecek olası ekstra `onDetect` çağrılarının aynı barkodu iki kez `onDetected` ile göndermesini engelleyen ucuz bir güvenlik önlemi.
