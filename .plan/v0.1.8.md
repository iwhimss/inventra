## Kök Neden Analizi

### [F2] Cihaz İsimlendirme

**Mevcut durum:** `sync_manager.dart` → `_getDeviceName()`, cihaz adını **platforma göre sabit bir string** olarak üretiyor:
```dart
if (Platform.isAndroid) return 'Android Cihaz';
if (Platform.isIOS) return 'iPhone / iPad';
if (Platform.isWindows) return 'Windows PC';
```
Bu isim, eşleştirme isteğiyle birlikte sunucuya gönderilip `paired_devices.device_name` olarak kaydediliyor ve **hiçbir zaman değiştirilemiyor**. İki Android telefon eşleştirildiğinde ikisi de "Android Cihaz" adıyla görünüyor — kullanıcının şikayeti tam olarak bu.

**Tasarım kararı:** v0.1.7'de belirlenen ilkeyle tutarlı olarak (kritik/yönetimsel ayarlar web dashboard'da, uygulama sürümünden bağımsız), yeniden adlandırma **web admin paneline** eklenir — uygulama tarafında hiçbir değişiklik gerekmez, cihaz eşleştirme akışı da değişmez (varsayılan isim yine platforma göre üretilir, sadece artık admin panelinden **düzenlenebilir**).

**Uygulama:** `/admin/devices` sayfasındaki onaylanmış cihazlar tablosuna, her satıra bir "Yeniden Adlandır" mini-formu eklenir (cihaz adı hücresi tıklanabilir bir input+kaydet butonuna dönüşür). Yeni `POST /admin/devices/rename` endpoint'i `paired_devices.device_name` sütununu günceller.

### [F3] Sepete Ekleme Sesi

**Araştırma bulgusu:** Bu özellik **zaten koda gömülü** — `sound_service.dart` içinde `playCartAdd()` metodu ve `cart_add.wav` asset dosyası mevcut; `pos_screen.dart`'ta sepete ürün eklemenin **tek merkezi noktası** olan `_showPriceSelectionDialog()` fonksiyonu her çağrıldığında `_postAddProduct()` → `SoundService.playCartAdd()` çalışıyor. Bu fonksiyon hem normal ürüne tıklama hem barkod tarama (kamera + USB okuyucu) hem paylaşılan barkod seçimi olmak üzere **sepete ürün eklenen tüm yollardan** çağrılıyor — yani "barkod taratınca da çalsın" isteği kod düzeyinde zaten karşılanmış durumda.

**Bulunan gerçek hata:** `sound_service.dart` → `_play()` metodunda **tüm ses kategorileri arasında paylaşılan tek bir debounce değişkeni** var:
```dart
static DateTime? _lastPlayTime;
...
if (_lastPlayTime != null && now.difference(_lastPlayTime!) < const Duration(milliseconds: 100)) return;
```
Bu, "aynı sesin 100ms içinde tekrar çalınmasını" değil, **hangi kategoriden olursa olsun HERHANGİ bir sesin 100ms içinde ardışık çalınmasını** engelliyor. Örneğin bir hata sesi (`playError`) çaldıktan hemen sonra (100ms içinde) bir sepete-ekleme sesi tetiklenirse, ikincisi **sessizce yutuluyor**. Bu, kullanıcının "bazen çalıyor bazen çalmıyor gibi" hissettiği asıl teknik nedendir.

**Çözüm:** Debounce değişkeni kategori bazlı yapılır (`Map<SoundCategory, DateTime>`), böylece bir kategorideki ses diğerini asla bastıramaz. Bu düzeltmeden sonra kullanıcıdan tekrar test etmesini rica edeceğiz — kod incelemesiyle başka bir engelleyici neden bulunamadı, mekanizma teknik olarak eksiksiz görünüyor.

Bu yüzden [B1] ve [B2] birlikte, [B2] önce olacak şekilde uygulanacak.

### [B2] Barkod Çerçeveden Çıkarılınca Tarama İptal Olmuyor — Kök Neden

**Mevcut mantığın varsayımı:** `_onDetect(BarcodeCapture capture)` her kamera karesinde çağrılır ve barkod artık görünmüyorsa `capture.barcodes` **boş bir liste** ile çağrılacağı varsayılıyordu:
```dart
void _onDetect(BarcodeCapture capture) {
  if (capture.barcodes.isEmpty) {
    _cancelCandidate();  // ← Bu callback hiç tetiklenmeyebilir
    return;
  }
  ...
}
```

**Gerçek kök neden:** `mobile_scanner` paketinin alt katmanı (Android'de ML Kit, iOS'ta Vision) barkod bulunamayan karelerde `onDetect` callback'ini **her zaman boş listeyle çağırmayabilir** — birçok platform implementasyonu, performans/pil optimizasyonu için barkod bulunamadığında callback'i **hiç tetiklemez** (ne dolu ne boş — sessizlik). Bu durumda kodumuzun `capture.barcodes.isEmpty` kontrolüne dayanan iptal mekanizması **hiçbir zaman çalışma şansı bulamıyor** — çünkü tetiklenmesi gereken callback'in kendisi hiç gelmiyor. Kullanıcının tarif ettiği "barkodu çerçeveden çıkardım ama tarama kesilmedi, son gördüğünü taratmaya devam etti" davranışı tam olarak bununla açıklanıyor.

**Çözüm — "son görülme" bekçi zamanlayıcısı (watchdog):** İptal kararını yalnızca gelen callback'lere bağımlı bırakmak yerine, **bağımsız bir zamanlayıcı** ile "bu aday barkodu en son ne zaman gerçekten gördüm" bilgisini takip edip, belirli bir süre (250ms) boyunca tazelenmezse **otomatik iptal** eder:

```dart
DateTime? _lastSeenAt;
Timer? _watchdog;

void _onDetect(BarcodeCapture capture) {
  if (capture.barcodes.isEmpty) { _cancelCandidate(); return; }
  final code = capture.barcodes.first.rawValue;
  if (code == null || code.isEmpty) { _cancelCandidate(); return; }

  _lastSeenAt = DateTime.now();
  _watchdog ??= Timer.periodic(const Duration(milliseconds: 100), (_) {
    if (_candidateCode != null &&
        DateTime.now().difference(_lastSeenAt!) > const Duration(milliseconds: 250)) {
      _cancelCandidate();
    }
  });

  if (_candidateCode != code) {
    setState(() => _candidateCode = code);
    _confirmController..stop()..reset()..forward();
  }
}
```
`_cancelCandidate()` içinde `_watchdog?.cancel(); _watchdog = null;` eklenir.

**Neden bu çalışır:** Artık iptal kararı "callback boş geldi" sinyaline değil, "son 250ms içinde bu barkodu gerçekten gördüm mü" sorusuna dayanıyor. Callback hiç gelmese bile (platformun sessiz kalma davranışı), zamanlayıcı bunu 250ms içinde fark edip iptal eder — bu, mekanizmanın asıl amacını (yanlış barkod fark edilirse yarıda kesmek) gerçekten yerine getirmesini sağlar.

### [B3] Sepetten Tek Ürün Silme — Onay Eksik

**Kök neden:** `pos_screen.dart`, sepet öğesi satırındaki "Sil" butonu doğrudan `cartNotifier.removeItem(item.productId)` çağırıyor — hiçbir onay adımı yok. Buna karşın sepeti toplu temizleyen "Sil" butonu (`_showTransferCartDialog` yanındaki, `Icons.delete_sweep`) zaten bir `AlertDialog` ile onay alıyor.

**Çözüm:** Tek ürün silme butonu, toplu silme ile **birebir aynı görsel/etkileşim deseniyle** bir onay dialogu gösterecek şekilde güncellenir (başlık: "Ürünü Sil", içerik: "[Ürün adı] sepetten çıkarılsın mı?", İptal/SİL butonları).
